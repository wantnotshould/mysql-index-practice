# 维护索引和表

即使用正确的数据类型创建了表并加上了合适的索引，工作也没有结束：还需要维护表和索引来确保它们都能正常工作。维护表有三个主要目的：
- 找到并修复损坏的表
- 维护准确的索引统计信息
- 减少碎片

## 找到并修复损坏的表

- 使用 `REPAIR TABLE` 对于 `MyISAM` 引擎表可以修复损坏的表
- 对于 `InnoDB` 表，可以使用 `ALERT TABLE` 命令来修复表的损坏：

    ```sql
    ALERT TABLE <table> ENGINE=INNODB;
    ```

*注*
- 修复表之前，最好先备份数据，以免意外数据丢失
- 修复 InnoDB 表时，一般会重建表的结构和数据，这样能够清理掉某些损坏的情况

## 更新索引统计信息

如果存储引擎向优化器提供的扫描行数信息不正确，或者执行计划本身太复杂以致无法准确地获取各个阶段匹配的行数，那么优化器会使用索引统计信息来估算扫描行数。MySQL 的优化器使用的是基于成本的模型，而衡量成本的主要指标就是一个查询需要扫描多少行。如果表没有统计信息，或者统计信息不准确，优化器就很有可能做出错误的决定。可以通过 `analyze table` 来重新生成统计信息，以解决这个问题。

可以使用 `show index from` 命令来查看索引的基数（cardinality）

## 减少索引和数据的碎片

### 索引碎片

B-tree 索引是 MySQL 中最常用的索引类型，但是随着数据的插入、更新和删除，B-tree 索引的结构可能会变得不连续，造成磁盘上的碎。

- **索引碎片**：随着插入、更新和删除的操作，B-tree 索引可能会出现空白节点或者索引页分裂，从而降低索引的效率

### 如何处理索引碎片？

`OPTIMIZE TABLE`：对于支持 `OPTIMIZE TABLE` 的存储引擎（如 **InnoDB** 和 **MyISAM**），可以通过执行该命令来优化表和索引。这将重新组织表和索引，减少碎片，提高查询效率

```sql
OPTIMIZE TABLE <table>;
```

使用 `ALTER TABLE`：如果存储引擎不支持 `OPTIMIZE TABLE`，可以通过执行一个不做任何实际操作的 `ALTER TABLE` 来重建表，这样可以清理碎片

```sql
ALTER TABLE <table> ENGINE=<engine>;
```
这个命令会重建表，清理索引碎片，并且重新分配存储空间

### 数据碎片

- 行碎片：指数据行被存储在多个磁盘页中。行碎片可能会影响到按主键访问数据的效率，导致查询延迟。
- 行间碎片：指逻辑上顺序的页或行，实际存储在磁盘上并不是顺序的，导致访问时需要随机磁盘I/O，影响查询性能。
- 剩余空间碎片：数据页中的空余空间可能没有被有效利用，造成大量无用的磁盘读写。

### 如何减少数据碎片？

- `OPTIMIZE TABLE`：同样，使用 OPTIMIZE TABLE 可以优化数据存储，减少碎片
- **数据导出再导入**：对于某些存储引擎，或者当 OPTIMIZE TABLE 无法有效优化时，可以通过将表数据导出到外部文件，再导入回数据库来重建表结构，从而清理碎片

    ```sql
    -- 导出数据
    mysqldump -u root -p <database> <table> > table_backup.sql;

    -- 删除原表
    DROP TABLE <table>;

    -- 导入数据
    mysql -u root -p <database> < table_backup.sql;
    ```
