# 前缀索引和索引的选择性

- **前缀索引 (Prefix Index) —— 手段**，对于 `BLOB`、`TEXT` 或者很长的 `VARCHAR` 类型列，必须使用前缀索引，因为 MySQL 并不支持对这些列的完整内容进行索引
  - 无法用于排序，无法用它来做 `GROUP BY` 或 `ORDER BY`
  - 无法用于覆盖索引,因为索引里只有前缀，MySQL 必须回表去拿完整字段
- **索引选择性（Index Selectivity）—— 标准** 它是用来判断：我这个前缀到底截取多长才合适？

*通过计算索引的选择性，来确定前缀索引的长度*


## 公式

索引选择性的计算公式非常简单：

$$Selectivity = \frac{Cardinality}{Total \text{ } Records}$$

即：
$$选择性 = \frac{不重复的记录数（基数）}{表总记录数}$$

- **Cardinality（基数）**：指该列中不同数值的数量（可以通过 `SHOW INDEX FROM table` 查看）
- **Total Records**：表中的总行数

**数值的含义**

选择性的取值范围在 $(0, 1]$ 之间：

- **选择性 = 1**：这是最理想的状态（例如主键索引或唯一索引）。每一行都有唯一的值，优化器可以瞬间定位到记录
- **选择性接近 0**：比如“性别”字段，只有“男”和“女”两个值，对于百万级数据来说，选择性极低
- **一般准则**：索引的选择性**越高**，MySQL 越倾向于使用它。如果选择性太低（例如低于 20%），MySQL 可能会认为全表扫描比走索引更快（因为回表操作的开销太大）

## 实战计算示例

假设你有一个 `users` 表，总行数为 **10,000** 行：

- **场景 A：`email` 字段**
    - 不重复的 email 数量为 9,500 个
    - 选择性 = $9500 / 10000 = 0.95$
    - **结论**：非常适合建索引

- **场景 B：`status` 字段（状态码：0, 1, 2）**
    - 不重复的值只有 3 个
    - 选择性 = $3 / 10000 = 0.0003$
    - **结论**：完全不适合单独建索引，建了也会被优化器忽略

[索引基础](./索引基础.md) 中的 `people` 表

```sql
-- 测试截取前 5 位字符的选择性
SELECT COUNT(DISTINCT LEFT(last_name, 5)) / COUNT(*) AS sel5 FROM people;

-- 测试截取前 10 位字符的选择性
SELECT COUNT(DISTINCT LEFT(last_name, 10)) / COUNT(*) AS sel10 FROM people;
```
