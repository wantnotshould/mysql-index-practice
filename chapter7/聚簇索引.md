# 聚簇索引（Clustered Index）

聚簇索引不是一种独立的索引类型，而是一种数据存储方式，它决定了数据在磁盘上的存储顺序。对于 **InnoDB** 存储引擎：
  - **数据与索引存储在同一个B树结构中**。
  - **一个表只能有一个聚簇索引**，因为数据本身是按照聚簇索引排序的。
  - 通常，**主键（PRIMARY KEY）** 就是聚簇索引。如果没有定义主键，InnoDB 会选择一个 **唯一非空索引** 作为聚簇索引。如果没有这样的索引，InnoDB 会 **隐式** 创建一个自增的主键作为聚簇索引

## 隐式主键

- 如果你没有定义主键，InnoDB 会自动生成一个隐式的主键（通常是 **自增** 的整数列），作为聚簇索引使用。
- **隐式主键的影响**：所有依赖此隐式主键的表都会依赖一个单一的自增值，可能导致较高的锁竞争和性能问题。

## 聚簇索引的优缺点

### 优点：

- **提高查询性能**：
  - 数据与索引在同一B树中，查询时可以直接访问数据，减少磁盘I/O
  - **相互关联的数据**（如电子邮件和用户ID）存储在一起，可以一次性读取相关数据
- **覆盖索引**：如果查询只涉及聚簇索引的字段，InnoDB 可以使用 **覆盖索引** 扫描，直接从索引中读取数据而不需要访问数据页

### 缺点：

- **插入性能**：数据插入时需要根据聚簇索引的顺序进行插入，因此如果数据插入顺序和主键顺序不一致，会导致页分裂（**Page Split**），影响性能
- **更新成本高**：如果更新聚簇索引列，InnoDB 需要将行移到新位置，这会造成性能开销
- **二级索引存储成本**：二级索引的叶子节点存储的是 **主键值**，因此访问二级索引时需要先查主键，造成两次索引查找

## 二级索引（Non-Clustered Index）

二级索引是指那些除了主键之外的索引。例如，若你对 `email` 列创建了索引，那么 `email` 上的索引就是二级索引
  - 二级索引的叶子节点不直接指向数据行，而是指向 **主键**
  - 查询时，**二级索引查找主键值**，然后再通过主键查找对应的数据行

### 示例

```sql
CREATE TABLE users (
    id INT NOT NULL AUTO_INCREMENT,  -- 主键列
    username VARCHAR(100),
    email VARCHAR(100),
    PRIMARY KEY(id)   -- 主键列，InnoDB 会根据此列创建聚簇索引
);

-- 创建二级索引
CREATE INDEX idx_email ON users(email);

-- 查询时通过二级索引查找
SELECT * FROM users WHERE email = 'bob@example.com';
```

**查询流程**：
- **通过二级索引 `idx_email` 查找 `email` 对应的主键（如 `id`）**。
- **通过主键 `id` 查找数据行**

## 页分裂（Page Split）

  当数据插入时，如果插入顺序不符合聚簇索引的顺序，InnoDB 可能会执行 **页分裂**，即将数据页分裂为两个页以容纳更多的数据

  **页分裂的影响**：会导致额外的磁盘空间使用，并且可能影响查询性能

### 示例

```sql
-- 插入数据时顺序不一致
INSERT INTO users (id, username, email) VALUES (1000, 'David', 'david@example.com');
INSERT INTO users (id, username, email) VALUES (500, 'Eva', 'eva@example.com');
```

*插入顺序与主键顺序不一致时，InnoDB 会尝试将数据插入已满的页中，这会导致页分裂操作，影响性能*

## 完整示例

假设我们创建一个 `users` 表，并在其上创建主键（聚簇索引）和二级索引

```sql
CREATE TABLE users (
    id INT NOT NULL AUTO_INCREMENT,  -- 主键列
    username VARCHAR(100),
    email VARCHAR(100),
    PRIMARY KEY(id)   -- 聚簇索引
);

-- 插入数据
INSERT INTO users (username, email) VALUES ('Alice', 'alice@example.com');
INSERT INTO users (username, email) VALUES ('Bob', 'bob@example.com');
INSERT INTO users (username, email) VALUES ('Charlie', 'charlie@example.com');
INSERT INTO users (username, email) VALUES ('David', 'david@example.com');

-- 创建二级索引
CREATE INDEX idx_email ON users(email);

-- 查询操作
SELECT * FROM users WHERE email = 'bob@example.com';
```

**聚簇索引**：`id` 是主键，数据会按 `id` 排序

**二级索引**：`email` 列上的二级索引会加速对 `email` 的查询，但查询时需要 **两次查找**：
- **通过二级索引** 查找 `email` 对应的主键（如 `id`）
- **通过主键查找数据行**


## `OPTIMIZE TABLE` 使用时机

*`回收空间`、 `减少碎片`、`提高查询效率`*

***注：***

- 锁表：`OPTIMIZE TABLE` 会对表进行 表级锁，在优化期间，其他查询会被阻塞，因此应当在数据库负载较低时执行，避免对生产环境的影响
- 时间成本：对于大表，执行 `OPTIMIZE TABLE` 可能需要较长时间，特别是在数据量很大的情况下，可能会影响数据库性能

不需要盲目的每天执行，可以通过以下 SQL 语句查看表的“碎片率”：

```sql
SELECT 
    TABLE_NAME, 
    DATA_FREE / 1024 / 1024 AS FREE_MB,  -- 碎片大小
    TABLE_ROWS,
    DATA_LENGTH / 1024 / 1024 AS DATA_MB -- 实际数据大小
FROM information_schema.tables 
WHERE TABLE_SCHEMA = '你的数据库名';
```

当 `DATA_FREE`（碎片空间）占据 `DATA_LENGTH`（总数据量）的 20% - 30% 以上时，执行优化才会有比较明显的收益

### 大量删除数据后

当你从表中删除大量数据时，虽然表中的数据被删除了，但 `InnoDB` 存储引擎并不会立即回收这些磁盘空间

```sql
DELETE FROM users WHERE last_login < '2020-01-01';
OPTIMIZE TABLE users;
```

### 大量更新数据后

当你更新表中的数据时，尤其是更新聚簇索引列（如主键），`InnoDB` 需要移动数据行，这可能导致数据页分裂（Page Split），表中可能会出现大量的空洞或者碎片

```sql
UPDATE users SET email = 'newemail@example.com' WHERE id = 123;
OPTIMIZE TABLE users;
```

### 频繁的插入操作后

如果你频繁地插入数据，并且这些数据并不是按照主键顺序插入，可能会导致 页分裂 和碎片化。定期执行 `OPTIMIZE TABLE` 可以整理表，使得数据更加连续，减少查询时的磁盘I/O

```sql
INSERT INTO users (username, email) VALUES ('Alice', 'alice@example.com');
INSERT INTO users (username, email) VALUES ('Bob', 'bob@example.com');
OPTIMIZE TABLE users;
```
