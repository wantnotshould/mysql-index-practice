# 索引基础

索引有很多种类型，可以为不同的的场景提供更好的性能。在 MySQL 中，索引是在存储引擎层而不是服务器层实现的。所以，并没有统一的索引标准：不同的存储引擎的索引的工作方式并不一样，也不是所有的存储引擎都支持所有类型的索引。即使多个存储引擎支持同一种类型的索引，其底层的实现也可以不同。*假设所有表都使用`InnoDB`引擎*

## 生效索引

### 全值匹配（Full Value Match）

查询条件包含索引中的所有列，索引效率最高，直接定位到具体记录

```sql
-- 精确匹配 last_name, first_name, dob 的值
SELECT * FROM people 
WHERE last_name = 'Smith' 
  AND first_name = 'Alice' 
  AND dob = '1990-05-15';
```

### 匹配最左侧（Matching the Leftmost Prefix）

只使用索引的第一列或前几列，只要包含了最左列 `last_name`，索引就能起作用

```sql
SELECT * FROM people WHERE last_name = 'Smith';
-- 或者
SELECT * FROM people WHERE last_name = 'Smith' AND first_name = 'Alice';
```

### 匹配列前缀（Matching a Column Prefix）

匹配某一列值的开头部分（仅限最左列或按顺序匹配到的列），这本质上是一个范围查询，索引依然有效

```sql
SELECT * FROM people WHERE last_name LIKE 'Smi%';
```

### 匹配范围（Matching a Range of Values）

在某一列上进行范围查找

```sql
SELECT * FROM people WHERE last_name BETWEEN 'Brown' AND 'Garcia';
```

### 精确匹配某一列而范围匹配另外一列

左侧列精确匹配，后续列范围匹配

*注：* 在 `first_name` 使用了范围查询后，其右侧的 `dob` 列将无法再利用索引进行过滤

```sql
SELECT * FROM people 
WHERE last_name = 'Smith' 
  AND first_name LIKE 'A%';
```

### 只访问索引的查询（Covering Index）

查询的字段都在索引树中，无需回表（回主键索引查其他字段），`EXPLAIN` 结果的 `Extra` 列会出现 `Using index`

```sql
-- 只需要索引里的这三列，不需要 SELECT *
SELECT last_name, first_name, dob FROM people WHERE last_name = 'Smith';
```

## 失效索引

### 违背最左匹配原则

如果跳过第一列 `last_name`，索引将完全失效

```sql
-- 失效：直接找名字，没法在按姓排序的索引里快速定位
SELECT * FROM people WHERE first_name = 'Alice'; 

-- 失效：只查生日
SELECT * FROM people WHERE dob = '1990-05-15';
```

### 中间跳过了某一列

中间列断开了，后面的列就无法使用索引进行过滤

```sql
-- last_name 走索引，但 dob 无法走索引（变成了全表扫描后的过滤）
SELECT * FROM people WHERE last_name = 'Smith' AND dob = '1990-05-15';
```

### 模糊查询前缀包含通配符

```sql
-- 失效：无法利用有序性
SELECT * FROM people WHERE last_name LIKE '%ith';
```

### 在索引列上进行函数计算或表达式

```sql
-- 失效：对 dob 使用了 YEAR() 函数
SELECT * FROM people WHERE YEAR(dob) = 1990;

-- 失效：对字符串进行了拼接
SELECT * FROM people WHERE CONCAT(last_name, first_name) = 'SmithAlice';
```

### 范围查询后的列失效

一旦某一列使用了范围查询（`>`, `<`, `BETWEEN`, `LIKE` `'A%'`），它右侧的所有列都无法利用索引的有序性

```sql
-- first_name 走了范围查询，dob 列在这里就不走索引排序了
SELECT * FROM people 
WHERE last_name = 'Smith' 
  AND first_name > 'Alice' 
  AND dob = '1990-05-15';
```

💡 小贴士：关于 `key_len`

**[EXPLAIN](https://dev.mysql.com/doc/refman/9.7/en/explain-output.html)** 结果中的 `key_len` 是一个关键的指标，过查看 key_len，你可以判断 SQL 到底利用了复合索引中的哪几列

- `last_name` (VARCHAR(50) utf8mb4): $50 \times 4 + 2$ (长度限制) + $1$ (NULL标志) = $203$
- `first_name` (VARCHAR(50) utf8mb4): $50 \times 4 + 2 + 1 = 203$
- `dob` (DATE): $3 + 1 = 4$
- **总和**：$203 + 203 + 1 = 407$（注：不同字符集长度计算略有差异，你的环境应该是 utf8mb4）
