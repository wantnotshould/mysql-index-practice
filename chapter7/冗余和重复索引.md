# 冗余和重复索引

MySQL 允许在相同列上创建多个相同的索引，虽然 MySQL 会抛出一个警告，但是并不会阻止你这么做。MySQL 需要单独维护重复的索引，优化器在优化查询的时候也需要逐个地进行评估，这会影响性能，同时也浪费磁盘空间。

重复索引是指在相同的列上按照相同的顺序创建的相同类型的索引。应该避免创建这样的重复索引，发现以后应该立即移除

*在 MySQL 中，重复索引（Redundant Indexes）和冗余索引（Duplicate Indexes）就像是给家里的门装了五把一模一样的锁——不仅没让房子更安全，反而让你每次开门都要试五次钥匙*

## 冗余索引（Redundant Indexes）

❌ 反面典型：最左前缀冗余

一个索引是另一个索引的前缀（Prefix）

```sql
CREATE TABLE redundant_example (
    id INT PRIMARY KEY,
    state CHAR(2),
    city VARCHAR(50),
    -- 索引 A
    INDEX idx_state_city (state, city),
    -- 索引 B：它是索引 A 的左前缀，属于冗余索引
    INDEX idx_state (state)
);
```

索引 A `(state, city)` 完全可以覆盖索引 B `(state)`· 的查询需求。MySQL 在执行 `WHERE state = 'NY'` 时，依然可以使用索引 A。因此，索引 B 是多余的

✅ 正面典型：非冗余场景

```sql
-- 后缀不同
INDEX idx_state_city (state, city),
INDEX idx_state_zip (state, zip)
-- 这不冗余，因为 city 和 zip 不同，各有用途

-- 扩展索引（针对不同类型）
INDEX idx_user (user_name),
FULLTEXT INDEX ft_idx_user (user_name)
-- 这不冗余，一个是普通 B-Tree 索引，一个是全文索引，用途完全不同
```

### InnoDB 特有的“隐藏”冗余

❌ 反面典型：主键冗余

```sql
CREATE TABLE innodb_redundant (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    -- 冗余！InnoDB 的二级索引自动包含了主键 id
    INDEX idx_name_id (name, id) 
);
```

如果你定义 `INDEX(name)`，InnoDB 在物理存储上其实已经存成了 `(name, id)`。手动加上 `id` 只是浪费空间

## 重复索引（Duplicate Indexes）

❌ 反面典型

```sql
CREATE TABLE bad_example (
    id INT PRIMARY KEY,
    user_name VARCHAR(50),
    -- 重复索引：PRIMARY KEY 已经包含了唯一索引
    UNIQUE (id),
    -- 重复索引：完全一致的列和顺序
    INDEX idx_user_name (user_name),
    INDEX idx_user_name_2 (user_name)
);
```

当你插入一行数据时，MySQL 必须同时更新 `PRIMARY KEY`、`UNIQUE(id)`、`idx_user_name` 和 `idx_user_name_2`。这会让写操作（INSERT/UPDATE/DELETE）变得异常缓慢
